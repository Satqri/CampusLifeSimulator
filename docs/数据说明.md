# 大学生活模拟事件系统 - 属性、隐藏变量与结算规范

本文档以当前代码实现为准，用于说明事件 JSON、普通交互和结局系统如何修改玩家状态。

## 一、核心语义

### 1. 可见属性

| 变量 | 范围 | 语义 | 设计方向 |
| --- | --- | --- | --- |
| `energy` | 0~100 | 体力。日常行动、学习、打工、冲刺会消耗；吃饭、睡觉、休息恢复。 | 单次降到 0 不会立刻结束；连续低体力多天才触发崩溃退学。 |
| `health` | 0~100 | 身体状态显示值。它是隐藏 `healthIndex` 的 0~100 截断投影。 | 不直接作为长期健康来源。 |
| `gold` | 0~9999 | 金钱。购买、赔差价消耗；打工获得。 | 支付型选项必须先做 `gold >= cost` 检定。 |
| `san` | 0~100 | 心态稳定度 / SAN。**越高越好**。 | 压力等价为 `stress = 100 - san`；SAN 为 0 触发崩溃退学。 |
| `academic` | 0~100 | 知识、学业能力、技术判断。 | 学习、阅读、考试、大创推进提升。 |
| `social` | 0~100 | 社交、表达、人际协调能力。 | 合作、社团、得体沟通提升。 |

> 重要：JSON 中的 `delta.san` 表示 SAN 变化，不表示压力变化。压力上升应写成 `san` 负数；放松、吃饭、睡觉、战斗胜利应写成 `san` 正数。

### 2. 压力换算

| 叙事表述 | 代码判定 |
| --- | --- |
| 压力很低、心态稳定 | `san >= 70` 或 `stress <= 30` |
| 压力可控 | `san >= 50` |
| 压力偏高 | `san < 40` |
| 心态危险 | `san < 30`，可能生成情绪敌人 |
| 崩溃 | `san <= 0` |

`SettlementResolver` 中的 `stress` 条件会换算为 `100 - san`，因此结局 JSON 可以继续使用 `stress`，但事件 JSON 推荐直接使用 `san`。

## 二、隐藏变量

隐藏变量保存在 `HiddenMap` 中，由 `hidden_delta` 修改。

### 1. 合并规则

| 类型 | 规则 |
| --- | --- |
| 整数计数 / 分值 | 默认累加，如 `friendBond: 1` 表示当前值 +1。 |
| 布尔值 | 覆盖，如 `sharedNotes: true`。 |
| 字符串 | 覆盖，如 `innovationResult: "excellent"`。 |
| 阶段变量 | 覆盖而不是累加：`friendStage`、`clubStage`、`innovationStage`。 |
| 事件元数据 | `_event_*_count` 累加，`_event_*_last_day` 覆盖。 |

### 2. 重要边界

| 变量 | 边界 |
| --- | --- |
| `healthIndex` | 0~130 |
| `innovationProgress` | 0~100 |
| `innovationDefenseScore` | -30~100 |
| `innovationTeamTrust` | -20~60 |
| `teacherTrust` | -20~40 |
| `friendBond` | -10~20 |
| `storeTrust` | -10~15 |
| `clubContribution` / `clubRelation` | -10~25 |
| `clubShowcaseScore` | -20~40 |
| `lateNightLevel` / `gameAddiction` | 0~60 |
| `friendStage` / `clubStage` | -1~4 |
| `innovationStage` | -1~5 |
| `*Count` | 0~99 |

## 三、统一结算顺序

所有普通交互、事件节点、睡觉、调试面板和按键 SAN 冲击测试都必须走统一后置检查：

1. 推进时间。
2. 应用可见属性 `delta`。
3. 应用隐藏变量 `hidden_delta`。
4. clamp 可见属性和隐藏变量。
5. 用 `healthIndex` 同步可见 `health`。
6. 检查即时结局：`healthIndex <= 0` 优先触发因病休学；`san <= 0` 或 `lowEnergyDays >= 3` 触发崩溃退学。
7. 如果第 14 天已结束，执行最终结算。
8. 未触发结算时才显示普通活动结果、继续事件跳转或触发后续时间事件。

## 四、数值标定原则

### 1. 常规活动

| 活动类型 | 推荐变化 |
| --- | --- |
| 连续重复同一活动 | 第 2 次起正面收益递减、负面效果递增；计数在睡觉后重置。 |
| 短休、吃饭 | `energy +5~30`，`san +2~10` |
| 睡觉 | 按睡眠时长恢复 `energy` 和 `san`，并做每日健康检查 |
| 学习 / 阅读硬核资料 | `academic +4~10`，`energy -5~-12`，`san -2~-6` |
| 文学阅读 / 放松娱乐 | `san +2~10`，轻微消耗或恢复 `energy` |
| 锻炼 | `energy -10~-15`，`san +3~8`，`healthIndex +5` |
| 夜班 / 加班 | `gold +30~80`，`energy -16~-25`，`san -4~-10`，`healthIndex -1~-3` |
| 垃圾食品 / 提神饮料 | `energy +5~15`，`san +2~8`，`healthIndex -1~-5` |

### 2. 事件结果

| 事件结果 | 推荐变化 |
| --- | --- |
| 成功解决、吃到好饭、关系变好 | `san +2~10` |
| 轻微尴尬或失败 | `san -2~-6` |
| 点名、社死、投诉、答辩失败 | `san -8~-20` |
| 熬夜攻关 | `energy -20~-30`，`san -10~-15`，`healthIndex -5` |
| 逃避高压责任 | 短期 `san +2~10`，但相关信任或进度下降 |

## 五、结局和称号

即时结局：

| 结局 | 条件 |
| --- | --- |
| 因病休学 | `healthIndex <= 0` |
| 崩溃退学 | `lowEnergyDays >= 3` 或 `san <= 0` |

最终结局按照 `assets/config/events/endings.json` 的 `priority` 从小到大执行。称号可叠加获得，条件读取隐藏变量和最终可见属性。

过分行为的承接规则：

| 行为 | 设计处理 |
| --- | --- |
| 连续逃课 | `skipClassCount >= 6` 且学业或老师信任不足时触发留级危机。 |
| 沉迷游戏但身体/学业崩坏 | `gamer_burnout` 优先于 `gamer_legend`，防止坏状态被正面电竞结局覆盖。 |
| 健康电竞路线 | `gamer_legend` 要求沉迷值足够、低逃课、健康和学业不过线崩坏。 |
| 社交达人 | 除高 `social` 外，还需要朋友或社团关系作为真实社交证据。 |
| 人脉王 | 可由深朋友关系达成，也可由朋友关系 + 社团关系共同达成，避免只绑定单一事件链。 |
| 过度熬夜/夜班 | `lateNightLevel` 会在每日检查中扣长期健康和 SAN，严重时转向崩溃或病休。 |

## 六、维护要求

- 新增事件时，所有 `san` 变化必须按“SAN 越高越好”填写。
- 支付选项必须先用 `gold` 检定，不能依赖 clamp 兜底。
- 阶段推进必须写 `friendStage`、`clubStage`、`innovationStage`，系统会按赋值处理。
- 如果新增隐藏变量，需要同步更新 `clampHiddenInteger()` 的边界规则或明确允许无边界。
- 不要在代码中直接绕过统一结算长期修改属性；优先使用 `GameContext::runTimedActivityWithHidden` 或在修改后调用 `finalizeStateChange`。
