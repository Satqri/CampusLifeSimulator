# 校园生活模拟器 AI 编程事件规范（主文档）

> 用途：这份文档不是小组讨论稿，而是给 Claude / AI 编程工具直接使用的**事件设计与实现规范**。目标是让 AI 在补充事件、重构 JSON、调整隐藏变量、扩展结局时，有一份统一、可执行、可映射到现有项目的说明。
>
> 适用范围：`plan/` 策划整理、`assets/config/events/*.json` 事件编写、事件链扩展、结局/称号补全、相关代码重构。
>
> 旧稿保留说明：`plan/person1_event_content_plan_v2(1).md` 与 `plan/校园生活模拟器——属性、隐藏变量与结局设计.md` 保留为历史草稿，本文件是新的 AI 实作主规范。

---

## 1. 文档目标

AI 在这个项目里写事件时，常见问题不是“不会写文案”，而是：

1. 把**常规交互**误写成每次都弹选项的剧情事件；
2. 把**设计稿变量名**、**JSON 字段名**、**代码里的真实结构**混用；
3. 只补了某个事件，却没同步考虑隐藏变量、长任务链、主结局、称号之间的闭环；
4. 沿用过时的分工文案，而没有产出能直接落地给程序的规格文档。

因此，本文件要求 AI 输出内容时：

- 优先提供**结构化规格**，而不是散文化脑暴；
- 明确区分：常规交互 / 突发事件 / 长任务链 / 结局结算；
- 允许基于新设计**调整当前 JSON 和代码结构**，但改动必须成体系，而不是只局部补丁；
- 任何新增事件都要能回答：
  - 它在哪里触发？
  - 它为什么不是普通交互？
  - 它改哪些基础属性？
  - 它改哪些隐藏变量？
  - 它是否影响某条任务链、某个结局、某组称号？

## 2. 项目真实架构约束

即使后续允许重构，也要理解当前项目是怎么组织的，避免脱离实现现实。

### 2.1 现有架构核心

- 游戏主循环与全局状态在 `main.cpp` 一侧集中管理；
- 事件系统由 `assets/config/events/*.json` 驱动；
- JSON 事件会通过 `EventRunner` 加载与执行；
- 当前只有少量传统交互逻辑保留，绝大多数内容应优先走 JSON 事件框架；
- 时间推进、上课时间检查、交互点触发、主结局和称号结算，本质上都是事件系统的外部驱动条件。

### 2.2 关键参考文件

- 事件格式规范：`docs/events_schema.md`
- 变量与边界说明：`docs/数据说明.md`
- 项目架构约束：`CLAUDE.md`
- 当前事件样例：
  - `assets/config/events/class_attendance.json`
  - `assets/config/events/friend_help_chain.json`
  - `assets/config/events/club_chain.json`
  - `assets/config/events/innovation_chain.json`
  - `assets/config/events/store.json`
  - `assets/config/events/endings.json`
  - `assets/config/events/titles.json`

### 2.3 允许的改动边界

用户已经明确：**当前已实现代码和 JSON 只是参考，可以更改当前代码和 JSON 的结构。**

因此，AI 设计时可以：

- 调整事件拆分方式；
- 重命名或重组事件链；
- 把一部分旧 JSON 改成新的统一结构；
- 把当前分散在多个文件里的逻辑收束成更清晰的事件/结算体系；
- 重新定义更适合 AI 生产的模板和字段约束。

但仍建议：

- 尽量复用已有变量名、事件 ID 风格、action_id 风格；
- 不无故发明第二套同义命名；
- 若要重构，先保证**文档层面的统一规则**成立，再推进代码和 JSON 迁移。

## 3. AI 写事件时的总原则

### 3.1 先判断事件类型，再写内容

所有内容先分四类：

| 类型 | 是否常驻 | 是否需要玩家选项 | 典型用途 |
| --- | --- | --- | --- |
| 常规交互 | 是 | 通常否 | 吃饭、睡觉、读书、锻炼、兼职 |
| 突发事件 | 否 | 是 | 打断计划、制造风险与分歧 |
| 长任务链节点 | 分阶段出现 | 是/否都可能 | 多次选择累积后果 |
| 结局/称号结算 | 学期末或强制触发 | 否 | 根据累计状态收束结果 |

### 3.2 不要把“日常动作”全部剧情化

以下行为默认应设计为**常规交互**，而不是每次都弹 choice：

- 食堂打饭 / 吃饭
- 宿舍睡觉 / 短休 / 学习 / 打游戏 / 刷手机
- 图书馆读不同类型的书
- 教室听课 / 看黑板 / 复习笔记
- 健身房跑步 / 力量训练 / 前台兼职
- 便利店买食物 / 买饮料 / 深夜泡面 / 收银兼职
- 大创日常推进（写方案、做 PPT、开会）

### 3.3 真正需要 choice 的内容

以下情况才优先做成事件选项：

- 行动被突发情况打断；
- 不同选项会导向不同的人际后果；
- 存在明显的属性门槛；
- 存在高风险高收益与低风险低收益分支；
- 该节点属于长任务链关键阶段；
- 该节点会改变后续结局判定走向。

### 3.4 新事件必须带着闭环意识来写

新增事件后至少检查：

1. 是否引入了新隐藏变量？
2. 是否已有旧变量可复用？
3. 是否会改变已有结局条件？
4. 是否需要新增称号、或修改现有称号触发条件？
5. 是否应该挂在现有链条上，而不是另起一条孤立路线？

## 4. 推荐事件数据模型

当前仓库里的 JSON 结构可继续使用，也可按以下方向统一升级。AI 生成时优先遵循这一层概念模型。

### 4.1 顶层事件结构

```json
{
  "id": "event_id",
  "trigger": { "type": "interaction", "action_id": "campus_square" },
  "root": "intro",
  "steps": {
    "intro": { "type": "display", "title": "...", "body": "...", "then": "decide" },
    "decide": { "type": "choice", "options": [...] }
  }
}
```

### 4.2 节点类型

建议统一使用以下五类：

- `display`：展示剧情文本
- `choice`：玩家做选择
- `check`：按状态、地点、标记判定
- `random_check`：按概率判定
- `outcome`：应用结果并结束或跳转

> 若未来要增强系统，也可以扩展中间节点类型，但不要轻易破坏这五类的基础语义。

### 4.3 属性与结果字段

推荐统一保留两层修改：

- `delta`：基础属性变化
- `hidden_delta`：隐藏变量/布尔/枚举变化

推荐规则：

- `delta` 只写基础可见属性：`energy`, `health`, `gold`, `san`, `academic`, `social`
- `hidden_delta` 写隐藏变量、布尔标记、枚举状态、链条计数
- 如果后续重构代码，优先让事件系统**正式支持 `hidden_delta` 作为一等字段**，而不是把隐藏变量更新散落在代码补丁里

### 4.4 条件写法建议

推荐支持并统一这几类条件：

- 属性判定：`stat`
- 地点判定：`location`
- 布尔/枚举判定：`flag` / `enum`
- 组合判定：`group`
- 元规则说明：`meta`（只用于文档或结算规则，不建议在运行时直接判）

### 4.5 trigger 建议

当前已存在的触发方式：

- `interaction`
- `time_schedule`

推荐保留并拓展，但命名上不要新旧混乱。

当前仓库已有代表性 action_id / method：

| 类型 | 值 |
| --- | --- |
| `action_id` | `campus_square`, `campus_bulletin`, `cafeteria_counter`, `cafeteria_table`, `library_desk`, `gym_bench`, `dormitory_room`, `dormitory_phone`, `store_counter`, `store_night_shift`, `store_shelf`, `store_register` |
| `method` | `crossed_class_time` |

如果未来要补更多交互点，建议沿用“地点_功能”的命名风格。

## 5. 术语统一规则

### 5.1 stress / san 统一

策划稿里经常写 `Stress`，但当前项目中：

- 旧 JSON 常用 `san`
- 结局/部分文档有时写 `stress`

AI 生成内容时建议采用以下规范：

- **文档层解释**：写“压力/心态（内部字段建议统一为 `san` 或统一迁移为 `stress`）”
- **若继续兼容现有 JSON**：优先沿用当前事件文件中的字段风格
- **若发起统一重构**：把整个项目一次性统一，不要半数写 `san` 半数写 `stress`

### 5.2 金钱上限冲突

旧稿存在 `gold 0~999` 与 `gold 0~9999` 两种口径。

建议本项目后续统一成一条明确规则，并在代码与文档同步：

- 如果强调“14 天短流程”，可选 `0~999`
- 如果强调“系统通用性/后续扩展”，可选 `0~9999`

AI 进行结构性重写时，要把这个上限当成**全局约束项**单独确认，不要默默沿用冲突值。

### 5.3 变量命名统一

隐藏变量统一使用：

- 小驼峰：`teacherTrust`, `friendBond`, `innovationProgress`
- 布尔值以状态含义命名：`innovationJoined`, `sharedNotes`, `owedFavor`
- 枚举值统一显式字符串，不要混成数字 magic number

## 6. 常规交互设计规范

常规交互的目标是：

- 让玩家主动走到地点执行动作；
- 不打断节奏；
- 提供稳定的资源转换；
- 为突发事件和长任务链提供基础舞台。

### 6.1 常规交互模板

```text
交互 ID：
交互点：
触发地点：
可用时段：
基础属性变化：
隐藏变量变化：
反馈文本：
是否推进时间：
是否可重复：
备注：
```

### 6.2 推荐常规交互组

#### 食堂
- 打饭窗口
- 餐桌吃饭

#### 宿舍
- 睡觉
- 短休
- 宿舍学习
- 打游戏
- 刷手机（可依附在床/桌）

#### 图书馆
- 理工书架
- 文史书架
- 参考资料
- 校史杂谈
- 阅读桌
- 图书管理员兼职

#### 教室
- 课桌听课
- 看黑板
- 复习笔记

#### 健身房
- 跑步机
- 力量训练
- 前台兼职

#### 便利店
- 买快捷食物
- 买提神饮料
- 深夜泡面
- 收银兼职

#### 大创日常推进
- 写方案
- 做 PPT
- 开会

### 6.3 AI 编写常规交互时的注意点

- 不要给每个常规动作都做 3~4 个选项；
- 反馈文本应短、稳、可重复阅读；
- 单次收益不宜过于戏剧化；
- 如果某个常规交互“偶尔出事”，请拆出独立突发事件。

## 7. 突发事件设计规范

突发事件承担三个功能：

1. 打断稳定资源循环；
2. 让属性门槛有戏剧化用途；
3. 把日常行为与隐藏变量、任务链连接起来。

### 7.1 突发事件模板

```text
事件 ID：
事件名：
所属场景：
触发条件：
推荐 trigger：
事件描述：
选项 A：
- 门槛：
- 基础属性变化：
- 隐藏变量变化：
- 结果反馈：
选项 B：...
后续影响：
```

### 7.2 选项设计原则

每个突发事件优先满足：

- 1 个普通选项
- 1 个属性门槛选项
- 1 个高风险高收益选项
- 1 个保底但不体面的选项

### 7.3 当前推荐保留的核心突发事件

| 场景 | 代表事件 | 作用 |
| --- | --- | --- |
| 教室 | 老师突然点名 / 老师让你上台写题 | 课程压力、老师信任、补救路径 |
| 食堂 | 最后一份鸡腿 | 小型社交分歧与资源取舍 |
| 图书馆 | 座位争夺战 | 学习效率与社交处理 |
| 宿舍 | 宿舍断网 / 社死朋友圈误发 | 宿舍内容的戏剧化入口 |
| 健身房 | 器械被占 | 小型社交与计划打断 |
| 校园 | 路演被拉上台 | 社交展示与尴尬处理 |
| 便利店 | 遇到老师 / 临时加班 / 夜宵售罄 / 顾客投诉 | 深夜路线、兼职路线、人情与压力 |

### 7.4 AI 对突发事件的输出要求

新增突发事件时，不要只写文案。必须同时回答：

- 它服务哪条资源循环？
- 它主要考验哪个属性？
- 它改动了哪些隐藏变量？
- 它有没有接入现有长任务链？
- 它是否值得保留为“高频随机事件”，还是更适合作为一次性链条节点？

## 8. 长任务链设计规范

长任务链是这个项目区别于普通加点模拟器的核心。

### 8.1 长任务链定义

长任务链不是“一次剧情事件”，而是：

- 前面选择留下状态；
- 后面节点读取这些状态；
- 最终生成副结局、称号、主结局修正或额外文本。

### 8.2 长任务链模板

```text
任务链名：
链条目标：
阶段表：
- 阶段名 / 事件 ID / 触发建议 / 关键门槛
关键变量：
- 哪些变量负责记录过程
阶段间继承规则：
- 前一个节点如何影响后一个节点
最终结果：
- 副结局 / 主结局影响 / 称号
```

### 8.3 推荐保留的四条核心链

#### 1）课程与点名链
核心问题：你对“上课/逃课/补救”的态度是什么？

建议保留的变量：
- `skipClassCount`
- `classAttendCount`
- `teacherTrust`
- `rollCallCount`
- `rollCallSavedCount`
- `absencePenaltyCount`
- `returnClassCount`
- `lateCount`

建议阶段：
- 上课时间到
- 是否在教室
- 是否触发点名
- 是否找同学救场
- 是否赶回补救
- 学期末副结局/称号结算

#### 2）同学互助链
核心问题：人情关系会不会在关键时刻回流？

建议保留的变量：
- `friendBond`
- `friendHelpCount`
- `friendRefuseCount`
- `friendRollCallHelpCount`
- `friendReviewCount`
- `sharedNotes`
- `owedFavor`

建议阶段：
- 卡题求助
- 食堂回礼
- 点名救场
- 考前互助复习
- 关系结算

#### 3）社团活动链
核心问题：你是参与者、协调者、主心骨，还是挂名成员？

建议保留的变量：
- `clubType`
- `clubActivityCount`
- `clubContribution`
- `clubRelation`
- `clubShowcaseScore`
- `clubShowcaseSuccess`

建议阶段：
- 招新
- 第一次任务
- 社团分歧
- 社团展示
- 学期末副结局/称号结算

#### 4）大创比赛链
核心问题：你是否真的做成了一件高压、高协作、高不确定性的事？

建议保留的变量：
- `innovationJoined`
- `innovationLeader`
- `innovationIntel`
- `innovationTopic`
- `innovationProgress`
- `innovationTeamTrust`
- `innovationDemoMode`
- `innovationSpeaker`
- `innovationDefenseScore`
- `innovationCrisisCount`
- `innovationResult`

建议阶段：
- 比赛通知
- 组队与选题
- 中期危机
- 答辩前夜
- 现场答辩
- 结果结算

### 8.4 设计要求

AI 在扩长任务链时应优先：

- 增加阶段间关联，而不是单纯增加节点数量；
- 让同一变量在多个系统之间回流，例如：
  - `teacherTrust` 同时影响课程与求助老师
  - `friendBond` 同时影响点名与考前复习
  - `innovationTeamTrust` 同时影响中期危机与答辩稳定性
- 每条链至少能产出 1 个结果维度：
  - 副结局
  - 称号
  - 主结局修正
  - 结算页附加文本

## 9. 主结局与称号的设计原则

### 9.1 主结局

主结局负责回答：这个 14 天学期，你最终成了什么样的人？

主结局应满足：

- 有明确优先级；
- 强制失败结局要能即时中断；
- 其他主结局要能覆盖不同玩法倾向；
- 不要被某条支线轻易完全覆盖。

当前推荐继续保留的主结局方向：

- 因病休学
- 崩溃退学
- 留级危机
- 创业新星
- 学霸
- 社交达人
- 打工皇帝
- 健身达人
- 人脉王
- 电竞高手
- 肝帝
- 摆烂快乐人
- 普通毕业

### 9.2 称号

称号不是主结局替代品，而是对玩家行为风格的补充描述。

规则建议：

- 每条链优先 1~3 个强识别度称号；
- 避免纯数值堆出来的大量重复称号；
- 称号文本要短，但结算说明可以更具体；
- 允许叠加，但展示数量要可控。

### 9.3 AI 写结局/称号时的检查清单

- 这个结局是否与其他结局重复表达？
- 这个称号是在描述玩家“能力”，还是“行为风格”？
- 它需要读取哪些变量？
- 如果新增了它，是否需要同步补文档索引和结算页展示逻辑？

## 10. AI 直接产出时的推荐模板

### 10.1 新增常规交互

```markdown
## 交互：library_browse_science
- 类型：常规交互
- 地点：图书馆
- 交互点：理工书架
- 时段：图书馆开放时间
- delta：academic +15, san +2, energy -12
- hidden_delta：libraryVisitCount +1, logicUnlocked = true
- 文案：你找到了和课程相关的资料，知识点开始连起来。
- 时间推进：30~60 分钟
- 备注：不做 choice
```

### 10.2 新增突发事件

```markdown
## 事件：random_library_seat
- 类型：突发事件
- trigger：interaction / library_desk
```
